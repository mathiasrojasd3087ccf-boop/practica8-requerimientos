# Informe Práctica 8
# Práctica 8 - Requerimientos e historias de usuario

Historias de usuario

Formato: *Como [rol], quiero [acción], para [beneficio]*

1. Como **estudiante**, quiero **subir mis prácticas de laboratorio a GitHub**, para **que el profesor pueda revisarlas fácilmente**.  
2. Como **profesor**, quiero **acceder al repositorio de cada estudiante**, para **verificar los avances de las prácticas**.  
3. Como **estudiante**, quiero **organizar mis prácticas en carpetas por semana**, para **tener un control claro de cada entrega**.  
4. Como **estudiante**, quiero **ver el historial de commits**, para **identificar los cambios realizados en cada práctica**.  
5. Como **profesor**, quiero **revisar las ramas creadas por cada estudiante**, para **asegurar el uso correcto de Git**.  
6. Como **estudiante**, quiero **crear ramas para experimentar con el código**, para **no afectar la versión principal**.  
7. Como **estudiante**, quiero **documentar cada práctica con README e informe**, para **cumplir con la evidencia requerida**.  
8. Como **profesor**, quiero **ver los commits con mensajes claros**, para **evaluar correctamente cada entrega**.  
9. Como **estudiante**, quiero **fusionar ramas sin perder información**, para **mantener un repositorio organizado**.  
10. Como **estudiante**, quiero **resguardar mis prácticas en GitHub**, para **evitar pérdida de información**.  

Requerimientos funcionales (40)

1. Crear repositorios por cada práctica.  
2. Subir archivos de código y documentos.  
3. Hacer commit de cambios en el repositorio local.  
4. Mostrar historial de commits.  
5. Crear ramas para desarrollo de nuevas funciones.  
6. Fusionar ramas correctamente.  
7. Resolver conflictos de fusión de ramas.  
8. Subir cambios al repositorio remoto.  
9. Documentar cada práctica con README.md.  
10. Documentar cada práctica con informe en Markdown.  
11. Permitir acceso al repositorio para revisión del profesor.  
12. Evidenciar trabajo individual de cada estudiante.  
13. Permitir actualización de archivos sin pérdida de información.  
14. Controlar versiones de los archivos subidos.  
15. Permitir revertir cambios si es necesario.  
16. Implementar mensajes de commit claros y descriptivos.  
17. Identificar autor de cada commit.  
18. Organizar archivos por carpetas por semana/práctica.  
19. Permitir clonación del repositorio en otra máquina.  
20. Facilitar la navegación entre ramas.  
21. Evidenciar avances de cada práctica.  
22. Permitir agregar nuevos archivos de forma incremental.  
23. Permitir eliminar archivos obsoletos.  
24. Mantener la integridad de los datos en todo momento.  
25. Evitar la sobreescritura accidental de archivos importantes.  
26. Permitir la revisión de cambios antes de fusionar ramas.  
27. Facilitar la colaboración si se trabaja en grupo.  
28. Registrar fecha y hora de cada commit.  
29. Permitir etiquetas (tags) para versiones importantes.  
30. Mantener historial completo de cada archivo.  
31. Permitir revertir a versiones anteriores si es necesario.  
32. Integrar evidencia de código con informes.  
33. Facilitar la exportación de informes.  
34. Permitir el seguimiento de errores o bugs en commits.  
35. Mantener seguridad de acceso al repositorio.  
36. Permitir integración con IDEs como NetBeans.  
37. Evitar duplicación de archivos.  
38. Facilitar búsqueda de archivos dentro del repositorio.  
39. Permitir subida de imágenes o diagramas relacionados con la práctica.  
40. Garantizar la trazabilidad completa de cada práctica de campo.  

Requerimientos no funcionales (2)

1. El sistema debe ser seguro y evitar pérdida de información.  
2. El sistema debe ser accesible desde cualquier computadora con internet y navegador.
